# Frontend Bug Fixes & Hardening — 2026-07-08

This document records a pass over every React component, Redux slice, and util in
`Frontend/src`, fixing logical bugs and defensive gaps that could crash the app or
cause unexpected behaviour. All changes are behaviour-preserving for the happy path;
they add safety for error/edge cases and correct a few genuine logic bugs.

---

## Critical — crash risks

### 1. `Components/Heading.js` — search suggestions could crash the render
The Google-suggest response is `[query, [suggestions], ...]`. When the backend
returns an error object (or an empty `[]`), `jsonData[1]` is `undefined`, so
`setSuggestions(undefined)` was called and the subsequent `suggestions.map(...)`
in JSX threw **"Cannot read properties of undefined (reading 'map')"**.

**Fix:** only store an array — `setSuggestions(Array.isArray(jsonData?.[1]) ? jsonData[1] : [])`.

### 2. `Components/CommentsContainer.js` — comments could crash the render
If the comments API returns an error object (e.g. comments disabled on a video),
`jsonData.items` is `undefined`; `setTOPComments(undefined)` then made
`TOPcomments.map(...)` throw.

**Fix:** guard with `Array.isArray(...)` and wrap the fetch in `try/catch`, falling
back to `[]`. Also added `videoId` to the effect dependency array so navigating
from one video to another **refetches** comments instead of showing stale ones.

### 3. `Components/SearchResults.js` — broken links & unstable keys for non-video results
The YouTube search API returns a mix of videos, channels, and playlists. For a
channel/playlist item `item.id.videoId` is `undefined`, producing
`/watch?v=undefined` links and duplicate `undefined` React keys.

**Fix:** filter results to items that actually have `id.videoId` before rendering,
guard `jsonData.items` with `Array.isArray`, and wrap the fetch in `try/catch`.

## Logical bugs

### 4. `Components/VideoContainer.js` & `Components/SearchResults.js` — loading skeleton never showed
State was initialised to `[]`, which is truthy, so the `videos ? results : skeletons`
ternary **always** took the results branch — the skeleton loader never appeared, and
a failed fetch left the grid stuck.

**Fix:** initialise to `null` (distinct "loading" state), set an array on success/error,
and wrap fetches in `try/catch`. Now: `null` → skeletons, `[]`/array → results.

### 5. `Utils/RandomMessageGenrator.js` — occasional blank chat message
`Math.round(Math.random() * chatMessages.length)` can evaluate to
`chatMessages.length`, an out-of-bounds index that returns `undefined`, so the live
chat intermittently rendered an empty message. (`RandomNameGenerator` already used
`Math.floor` correctly.)

**Fix:** switched to `Math.floor(...)` to keep the index in range.

### 6. `Components/Sidebar.js` — dead code reading non-existent state
`useSelector((store) => store.toggleMenu)` always returned `undefined` (the real path
is `store.appconfigslice.toggleMenu`), and both `isMenuOpen` and `dispatch` were
unused.

**Fix:** removed the dead selector/dispatch and their now-unused imports.

### 7. `Components/Login.js` & `Components/SignUpForm.js` — hardcoded `localhost` backend
Both forms called `http://localhost:3000/{login,signup}` directly, so auth silently
failed in any deployed build. `Utils/Constants.js` already derives a `BACKEND_URL`
from `REACT_APP_BACKEND_URL` for the search API.

**Fix:** exported `BACKEND_URL` from `Constants.js` and used it in both fetch calls.

# Follow-up: Search bar fixes

### Port conflict — backend & frontend on the same port
`react-scripts start` defaults to port **3000**, the same port the Express backend
listens on (`process.env.PORT || 3000`), so only one could run at a time. The backend
CORS config already whitelists `http://localhost:3001` as the frontend origin, so the
intended split is **backend :3000, frontend :3001**.

**Fix:** pinned the frontend dev server to 3001 in `Frontend/package.json`:
`"start": "PORT=3001 react-scripts start"`. The backend stays on 3000, which is what
`BACKEND_URL` in `Constants.js` already points to. No backend change needed.

### Keyboard support in the search box (`Components/Heading.js`)
Previously the only way to search was to click the 🔍 button; Enter did nothing and the
suggestion dropdown was mouse-only.

- **Arrow keys:** `↓`/`↑` move a highlight through the suggestion list (with wrap-around),
  tracked by a new `activeIndex` state. The highlighted item is styled, and hovering a
  suggestion syncs the highlight. `Esc` closes the dropdown.
- **Enter:** now runs the search. If a suggestion is highlighted it searches for that,
  otherwise it uses the typed text. Implemented via a new `onKeyDown` handler on the input.
- **Shared `runSearch(query)` helper:** updates the input, stores the query in redux,
  closes the dropdown and programmatically navigates to `/results` (via `useNavigate`).
  The 🔍 button and suggestion clicks both call it, replacing the old `showResultVideos`
  + `<Link>` wrapper. Suggestion clicks use `onMouseDown` so they fire before the input's
  `onBlur` hides the list.
